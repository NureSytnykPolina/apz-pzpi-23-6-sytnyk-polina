2.2 Опис прийнятих інженерних рішень
У межах роботи було розроблено мобільний застосунок для програмної системи дистанційної оренди речей Click_Rent. Серверна частина системи була реалізована раніше, тому мобільний застосунок створено як клієнтську частину, що взаємодіє з уже наявним backend API.
Мобільний застосунок реалізовано для платформи Android мовою Kotlin. Для побудови інтерфейсу використано XML-розмітку та стандартні компоненти Android SDK. Такий підхід дозволяє створювати зрозумілий інтерфейс користувача та спрощує подальшу підтримку застосунку.
Для організації функціональності застосунку використано архітектуру на основі Activity. Кожен екран відповідає за окремий сценарій роботи користувача. Для навігації між екранами використовуються Intent.
Для збереження інформації про користувачів, активну сесію та орендовані речі використовується SharedPreferences. Це дозволяє зберігати дані локально без використання окремої бази даних на мобільному пристрої.
Для взаємодії із серверною частиною використовується бібліотека OkHttpClient. Дані між мобільним застосунком і сервером передаються у форматі JSON через REST API.
У застосунку реалізовано функціональність реєстрації та входу користувача. Після успішної авторизації користувач отримує доступ до каталогу речей, перегляду інформації про товари та оформлення оренди.
Адміністративний функціонал у мобільний застосунок не включався, оскільки застосунок орієнтований на звичайних користувачів сервісу оренди.
2.3 Будова мобільного застосунку
Мобільний застосунок складається з таких основних частин:
1.	екрани інтерфейсу користувача;
2.	модулі взаємодії з REST API;
3.	локальне сховище SharedPreferences;
4.	моделі даних;
5.	навігаційна система між екранами.
У застосунку реалізовано екрани входу, реєстрації, головного меню, каталогу речей, перегляду товару та списку орендованих речей.
На головному екрані користувач може перейти до загального каталогу, жіночих речей, чоловічих речей або сторінки власних оренд.
На екрані каталогу відображаються доступні товари. Для демонстрації роботи системи реалізовано кілька категорій товарів, зокрема ноутбуки, камери, інструменти, жіночий та чоловічий одяг.
На сторінці товару користувач може переглянути назву речі, опис, вартість оренди та виконати оренду товару.
Екран «Моя оренда» дозволяє переглядати список орендованих речей, їх статус та очищати список оренд.
Також реалізовано можливість виходу з облікового запису та повернення на попередні екрани за допомогою кнопок навігації.
2.4 Взаємодія мобільного застосунку з системою
Мобільний застосунок взаємодіє з раніше реалізованою серверною частиною через REST API. Серверна частина відповідає за бізнес-логіку системи, роботу з базою даних, перевірку доступності товарів та обробку інформації про оренди.
Основний сценарій роботи має такий вигляд:
1.	Користувач виконує вхід або реєстрацію.
2.	Мобільний застосунок відкриває головне меню.
3.	Користувач переходить до каталогу речей.
4.	Застосунок отримує інформацію про товари через backend API.
5.	Користувач відкриває сторінку конкретного товару.
6.	Користувач оформлює оренду.
7.	Інформація про оренду зберігається локально та відображається на сторінці «Моя оренда».
8.	За необхідності користувач може очистити список орендованих речей.
Таким чином мобільний застосунок не дублює серверну логіку, а надає користувачу зручний інтерфейс для роботи з функціональністю системи Click_Rent.
2.5 UML-діаграми
UML діаграма прецедентів відображає основні можливості користувача в мобільному застосунку Click_Rent. На діаграмі представлено процеси реєстрації та авторизації користувача, перегляду каталогу речей, перегляду чоловічих та жіночих категорій, відкриття інформації про товар, оформлення оренди, перегляду списку орендованих речей та виходу із системи. Діаграма дозволяє визначити основні сценарії взаємодії користувача з мобільним застосунком.
 
Рисунок 2.1 — UML діаграма прецедентів мобільного застосунку
UML діаграма компонентів демонструє структуру мобільного застосунку та його взаємодію з іншими частинами системи Click_Rent. На діаграмі відображено основні компоненти застосунку, зокрема Activity-компоненти, локальне сховище SharedPreferences, API-клієнт OkHttpClient, серверну частину ASP.NET Core та базу даних SQL Server. Діаграма показує потоки обміну даними між мобільним клієнтом і backend-системою.

 
Рисунок 2.2 — UML діаграма компонентів мобільного застосунку
UML діаграма діяльності відображає послідовність дій користувача під час роботи із системою Click_Rent. На діаграмі показано процес запуску застосунку, виконання входу або реєстрації, відкриття каталогу речей, вибору товару, перегляду його характеристик, оформлення оренди та переходу до сторінки «Моя оренда». Також відображено можливість очищення списку орендованих речей та завершення роботи із застосунком.
 
Рисунок 2.3 — UML діаграма діяльності мобільного застосунку
UML діаграма пакетів відображає логічну структуру програмного коду мобільного застосунку. На діаграмі показано основні пакети проєкту: Activities, API, Storage та Layouts. Пакет Activities містить усі екрани застосунку, пакет API відповідає за взаємодію із серверною частиною, Storage забезпечує локальне збереження даних, а Layouts містить XML-файли інтерфейсу користувача. Такий поділ дозволяє забезпечити модульність проєкту та спрощує його подальшу підтримку й розвиток.
