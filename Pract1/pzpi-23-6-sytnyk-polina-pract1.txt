2 ЗАВДАННЯ
Метою практичного заняття є ознайомлення із шаблоном проєктування програмного забезпечення Observer, аналіз його призначення, структури, особливостей застосування та практичної реалізації засобами об’єктно-орієнтованого програмування.
Відповідно до поставленого завдання необхідно було підготувати доповідь на тему «Шаблон (патерн) проєктування ПЗ: Observer», обрати шаблон проєктування з книги GoF, крім Singleton, проаналізувати призначення, структуру, переваги та недоліки патерна Observer, розглянути типові ситуації, у яких доцільно застосовувати зазначений патерн, підготувати приклад програмної реалізації патерна, розробити слайди презентації українською мовою у науково-технічному стилі, забезпечити відповідність оформлення слайдів установленим вимогам щодо читабельності, лаконічності та нумерації, підготувати відеозапис доповіді тривалістю 10–12 хвилин, опублікувати відеозапис на платформі YouTube, створити GitHub-репозиторій для виконання завдань з дисципліни та оформити звіт у форматах PDF і TXT.

3 ОПИС ВИКОНАНОЇ РОБОТИ
3.1 Загальні відомості про шаблони проєктування
Шаблони проєктування програмного забезпечення являють собою типові перевірені рішення поширених задач проєктування, що виникають під час створення програмних систем. Їх використання дає змогу формалізувати підходи до побудови архітектури, підвищити гнучкість програмного коду, покращити його підтримуваність та забезпечити повторне використання вже напрацьованих рішень.
У класичній класифікації GoF шаблони поділяються на три основні групи: породжувальні, структурні та поведінкові. Патерн Observer належить до поведінкових шаблонів проєктування, оскільки визначає механізм взаємодії між об’єктами та забезпечує автоматичне інформування залежних компонентів про зміни стану.

3.2 Опис обраного патерна
Обраним для розгляду шаблоном проєктування є Observer.
Observer — це поведінковий патерн проєктування, який визначає залежність типу «один до багатьох», у межах якої один об’єкт (Subject) автоматично повідомляє всі залежні об’єкти (Observers) про зміну свого стану.
Основне призначення патерна полягає в організації механізму підписки та розсилання повідомлень між компонентами системи без жорсткої залежності між ними.
Передумовами застосування патерна є необхідність повідомляти багато об’єктів про одну подію, реалізувати систему підписок, забезпечити слабку зв’язаність між модулями та автоматизувати процес оновлення даних.
До переваг патерна належать зменшення зв’язності між компонентами, легкість розширення системи, підтримка принципу відкритості/закритості та повторне використання коду.
Недоліками є складність відстеження ланцюгів подій, можливі проблеми продуктивності при великій кількості спостерігачів та ризик каскадних оновлень.

3.3 Структура патерна Observer
Структурно патерн Observer складається з кількох основних елементів:
•	Subject (Видавець) — зберігає список спостерігачів та надає методи підписки й відписки;
•	Observer — інтерфейс, що визначає метод update();
•	ConcreteSubject — конкретна реалізація видавця;
•	ConcreteObserver — об’єкт, який реагує на зміни стану видавця.
Принцип роботи полягає в тому, що після зміни стану видавець викликає метод повідомлення, який послідовно звертається до всіх підписаних спостерігачів. Унаслідок цього всі залежні компоненти синхронізують власний стан.

3.4 Реалізація патерна Observer
Для демонстрації практичного застосування патерна було розглянуто приклад програми, у якій реалізовано механізм підписок і сповіщень.
Було створено інтерфейс IObserver, клас Subscriber, що реалізує метод Update(), а також клас Publisher, який зберігає список підписників і виконує їх сповіщення.
Після виклику методу Notify() кожен підписаний об’єкт отримує повідомлення та виконує відповідну дію.
Такий підхід забезпечує можливість легко додавати нових підписників без зміни логіки видавця.

3.5 Практичні приклади використання
Патерн Observer широко застосовується в сучасному програмному забезпеченні. Прикладами використання є:
•	підписки на канали в YouTube;
•	email-розсилки;
•	сповіщення в соціальних мережах;
•	обробка подій у графічних інтерфейсах;
•	системи моніторингу;
•	біржові системи оновлення курсів.

3.6 Підготовка презентації
У межах виконання роботи було створено презентацію, присвячену патерну Observer.
До презентації включено такі питання:
•	поняття шаблонів проєктування;
•	класифікація GoF;
•	сутність патерна Observer;
•	призначення та сфери застосування;
•	структура патерна;
•	переваги та недоліки;
•	приклад програмної реалізації;
•	практичні приклади використання;
•	висновки.

4 ВИСНОВКИ
У результаті виконання практичного заняття було розглянуто шаблон проєктування Observer як один із поведінкових патернів, наведених у класифікації GoF.
У ході роботи було проаналізовано призначення цього патерна, його основні структурні елементи, принцип функціонування, переваги, недоліки та особливості практичного застосування під час розроблення програмного забезпечення.
Було встановлено, що патерн Observer доцільно використовувати у випадках, коли система повинна автоматично повідомляти багато залежних об’єктів про зміну стану одного джерела даних. Такий підхід забезпечує слабку зв’язаність між компонентами, підвищує масштабованість і спрощує супровід програмних систем.
Також було розглянуто приклад програмної реалізації патерна, що дозволило наочно продемонструвати взаємодію між видавцем і спостерігачами.
Загалом виконане практичне заняття дало змогу поглибити знання з архітектури програмного забезпечення, зрозуміти доцільність застосування патернів проєктування та сформувати практичні навички використання патерна Observer під час побудови сучасних програмних систем.

Приклад програмного коду
interface IObserver
{
    void Update(string message);
}

class Subscriber : IObserver
{
    public void Update(string message)
    {
        Console.WriteLine("Отримано: " + message);
    }
}

class Publisher
{
    private List<IObserver> observers = new List<IObserver>();

    public void Subscribe(IObserver observer)
    {
        observers.Add(observer);
    }

    public void Notify(string msg)
    {
        foreach (var observer in observers)
        {
            observer.Update(msg);
        }
    }
}

class Program
{
    static void Main()
    {
        Publisher publisher = new Publisher();

        Subscriber sub1 = new Subscriber();
        Subscriber sub2 = new Subscriber();

        publisher.Subscribe(sub1);
        publisher.Subscribe(sub2);

        publisher.Notify("Нове повідомлення!");
    }
}
